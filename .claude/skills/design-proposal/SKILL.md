# Design Proposal

UIの構成・レイアウト・表記で複数の選択肢がある場合に、バリアントを作成してユーザーに視覚的に比較選択してもらうワークフロー。

## ワークフロー

### Phase 1: 提案（テキスト）

ユーザーの相談内容を分析し、2〜5個のバリアントをテキストで提案する。

各バリアントには以下を含める:
- **バリアント名**: 短い識別名（例: "Variant A: 現在→適用後"）
- **概要**: 1〜2文でアプローチを説明
- **メリット/デメリット**: 簡潔に

提案後、「全バリアントをモックアップとして作成しますか？」と確認する。

```
例:
---
**Variant A: 現在 → オーダー適用後**
カラムヘッダーで「現在」と「オーダー適用後」を明示。新規アイテムはダッシュ枠で空欄を表現。
- メリット: 時系列が明確
- デメリット: 新規のみの場合に空欄が目立つ

**Variant B: 適用前 → 適用後**
...（同様に記載）
---

全バリアントをモックアップとして作成しますか？
```

### Phase 2: バリアント作成

ユーザーが承認したら、全バリアントをHTMLファイルとして作成する。

#### フォルダ構成

元ファイルと同じディレクトリ内にサブフォルダを作成する。

```
元ファイル: mockups/order_flow/order-detail.html
バリアント: mockups/order_flow/{variants-folder}/
  ├── variant-a-{slug}.html
  ├── variant-b-{slug}.html
  ├── variant-c-{slug}.html
  └── variant-d-{slug}.html
```

- フォルダ名はバリアントの対象に応じて命名（例: `ba-variants/`, `layout-variants/`）
- ファイル名は `variant-{a|b|c|...}-{英語slug}.html`

#### 各バリアントファイルの構成

- 自己完結するHTMLファイル（CSSインライン、外部依存は元ファイルと同じCDNのみ）
- 先頭にバリアント説明ラベルを配置（どのバリアントか、特徴は何かが一目でわかるように）
- 対象セクションのみを抜粋表示（ページ全体を再現する必要はない）
- 同じサンプルデータを全バリアントで統一し、デザインの違いだけに集中できるようにする

### Phase 3: 選択と反映

ユーザーに全バリアントをブラウザで確認してもらい、採用するバリアントを選んでもらう。

選択されたバリアントのCSS・HTMLを元ファイルに反映する:
1. 元ファイルの対象セクションのCSSを、選択バリアントのCSSで置換
2. 元ファイルの対象セクションのHTMLを、選択バリアントのHTMLで置換
3. 不要になったCSSクラス（他バリアント固有のもの）を削除

バリアントフォルダは反映後も参考用として残す（ユーザーが明示的に削除を求めない限り）。

## 判断基準: いつこのスキルを使うか

以下の条件を満たす場合に能動的に提案する:

- デザイン上、2つ以上の有効なアプローチが考えられる
- テキスト説明だけでは違いが伝わりにくく、視覚的な比較が有効
- ユーザーが「どうしたらいい？」「いい方法ある？」と判断を求めている

以下の場合は使わない:

- 明らかに正解が1つしかない場合（バグ修正、誤字修正など）
- ユーザーが既に具体的な指示を出している場合
- 変更が軽微でバリアント作成のコストに見合わない場合
